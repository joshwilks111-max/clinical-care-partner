# v3.1 — thin-harness / fat-skill rewrite (review-pass summary)

The v3.1 architecture rewrite collapsed three serial routes (`/api/turn1`, `/turn1.5`, `/turn2`)
into a single chat route (`/api/chat`) driven by `streamText` and four tools, with the workflow
shape moved up the stack into a self-describing skill at `skills/dose-calculator/`. Four review
passes hardened the plan before any code was written: **`/plan-eng-review`** locked the SDK 6
shape (D1, D2 — `streamText` + `onFinish` walking `event.steps[].toolResults`, plan's bare
`{ text, toolCalls, toolResults }` destructure was wrong; confirmed against
`node_modules/ai/dist/index.d.ts`); **`/plan-design-review`** locked the Heidi-grammar 3-column
shell (D4, D13–D17 — full migration of the console grammar, `originalNote` pin server-derived to
defend against client-side-provenance-forgery, four design tokens, full a11y in v1);
**`/plan-devex-review`** locked the operator-side orchestration (D18–D20 — scripted fan-out,
per-lane `PROGRESS.md`, bounded fix-loop); **`/plan-ceo-review`** locked the safety posture (D3
refusal split across three layers, D7 two-layer safety framing in DESIGN.md, deploy-before-delete
in Phase 3, the rebase gate before Phase 1 fan-out, and the **Why not RAG** beat the brief
deserves — the brief licenses RAG but says "or similar", and the senior reading of "similar" is
*whatever retrieval mechanism is appropriate to the corpus size*). Four cleanup passes followed:
**Promptfoo removed** (single eval surface — the skill's `evals/cases.jsonl` is consumed by both
the skill's contract test and the harness's `app/api/chat/route.test.ts` integration test under
vitest, so two eval surfaces became one); **safety-canon removed** (the dev-time
`scripts/lint_skill_output.ts` lints the skill's output during iteration; the same regex guard
runs at runtime in `lib/response-validator.ts`, so the standalone safety-canon doc was redundant);
**clinical numbers stripped from `references/`** (`grep -E '\b[0-9]+(\.[0-9]+)?\s*(mg|ml|mcg|kg)\b'
skills/dose-calculator/references/*.md` returns 0 matches — the architectural promise that the
skill markdown carries no clinical content is now mechanically verifiable); and **verified
clinical content moved to the registry via `differential_check[]` + `severity_rows[].description`**
(the four croup must-not-miss differentials with their distinguishing features, and the four
severity-row descriptions from Bjornson & Johnson modified CMAJ 2013, now live in
`registry/guidelines.ts` as the single source of truth — the skill reads `differential_check[]`
in Phase 2 and matches the patient to a `severity_row.description` in Phase 3, so the skill never
needs to know what "moderate croup" looks like). The Sen et al. citation cited in DESIGN.md §6
("Why not RAG") was verified against arXiv (2605.15184, authors Sen / Kasturi / Lumer / Gulati /
Subbiah, submitted 2026-05-14) — it is the same paper already in `research/papers.md` (filed
under the working name "PwC grep"); rather than write a parallel entry, the existing entry was
**extended** with the full author list, the experimental setup (LongMemEval 116-question sample
across Chronos + Claude Code + Codex + Gemini CLI harnesses), and a "Why this matters for v3.1"
subsection that ties the inline-delivery finding explicitly to the `load_guideline` typed-registry
choice. **Lock count entering Phase 2:** 20 decisions (D1–D20). Net LOC delta vs v1.3 main:
**-1,970** (~+2,030 new − ~4,000 deleted). Final test count: ~87 (down from 222; the safety-critical
layer is better-covered because the validator + contract test are new).
